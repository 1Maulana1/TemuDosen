<objective>
Build the DRF read/write API over `SessionLogbook`: lecturer list + detail, approve-with-edits, manual-notes fallback (STT-07), and the student read-only endpoint (STT-06). All endpoints reuse the project's established ownership + state-machine-guard discipline (`CompleteSessionView`'s 404→403→400 sequence). This is pure CRUD over the Wave-1 model — it does not touch the Celery pipeline, so it runs in parallel with the service wrappers (06-03).

Purpose: The lecturer/student UI (Wave 4) and the integration test (Wave 6) call these endpoints. Approval is the pipeline's sole safety gate (AI-SPEC FM #5) — enforce ownership and the `ready_for_review → approved` transition server-side, never trusting the client.

Output: `serializers.py`, `views.py` (five view classes), `urls.py` (`logbook_urlpatterns`), a `config/urls.py` include, and endpoint tests covering ownership, state guards, and the manual-notes path.
</objective>

<artifacts_produced>
This plan produces:
- **`serializers.py`:** `LogbookListSerializer` (session/student/date/status/is_manual for S-12), `LogbookDetailSerializer` (transcript + summary_raw/edited + status + groundedness flags for S-13/S-15), `ApproveLogbookSerializer` (`serializers.Serializer` accepting edited `summary_edited` JSON), `ManualNotesSerializer` (accepts `notes` text)
- **`views.py`:** `LecturerLogbookListView` (GET, advisee-scoped), `LecturerLogbookDetailView` (GET one, ownership), `ApproveLogbookView` (POST — save summary_edited, status→approved, approved_at/by), `ManualNotesView` (POST — save notes on a failed logbook → done/approved-equivalent, `is_manual=True`), `StudentLogbookView` (GET — student's own approved logbook only)
- **`urls.py`:** `logbook_urlpatterns` (routes: `lecturer/`, `<int:session_id>/`, `<int:session_id>/approve/`, `<int:session_id>/manual-notes/`, `student/<int:session_id>/`)
- **`config/urls.py`:** `path('api/logbook/', include((logbook_urlpatterns, 'logbook')))`
</artifacts_produced>
