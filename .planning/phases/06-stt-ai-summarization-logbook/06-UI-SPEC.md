**Design system status:** No `components.json` anywhere in the repo — shadcn was never initialized in this project (confirmed Phases 1–5/8). ## SessionLogbook Status Badge Contract (cross-cutting — lecturer, student, admin all see the same language)

Extends `StatusBadge.tsx`'s existing `BadgeStatus` union with a **separate** logbook-status vocabulary — do not conflate with the existing queue-status badges (`MENUNGGU`/`DISETUJUI`/`BERLANGSUNG`/`SELESAI`/`DIBATALKAN`/`REVISI`/`DITOLAK`), which describe the **session/queue** lifecycle, not the **logbook-processing** lifecycle (D-06's `pending → transcribing → summarizing → ready_for_review → approved` / `failed`). A single session can show both badges side by side once it reaches `DONE` (e.g. "Selesai" + "Merangkum AI…").

| Status key (D-06) | Label (Bahasa) | Background / text token | Icon | In-progress? |
|---|---|---|---|---|
| `pending` | "Menunggu Diproses" | `bg-status-pending-bg text-status-pending-text` (amber) | `schedule` | No |
| `transcribing` | "Mentranskripsi…" | `bg-status-ongoing-bg text-status-ongoing-text` (teal) | `graphic_eq` + `animate-pulse` | **Yes** |
| `summarizing` | "Merangkum AI…" | `bg-status-ongoing-bg text-status-ongoing-text` (teal) | `auto_awesome` + `animate-pulse` | **Yes** |
| `ready_for_review` | "Siap Ditinjau" | `bg-status-approved-bg text-status-approved-text` (blue) | `rate_review` | No |
| `approved` | "Disetujui" | `bg-status-done-bg text-status-done-text` (green) | `check_circle` | No |
| `failed` | "Gagal Diproses" | `bg-status-cancelled-bg text-status-cancelled-text` (red) | `error_outline` | No |

Badge markup: same shape as existing `StatusBadge` (`rounded px-2 py-1 text-[11px] font-bold uppercase`), with the icon (16px, `text-sm`) prepended and a `gap-1` flex wrapper only for the two in-progress states — terminal states (`pending`, `ready_for_review`, `approved`, `failed`) keep the exact existing text-only badge shape, no icon, for visual consistency with every other status badge in the app.

**Accessibility:** in-progress badges use `role="status" aria-live="polite"` on their wrapping element (same pattern as `LecturerDashboard.tsx`'s existing "Merekam…" chip) — a lecturer polling this screen must get an announced update when `transcribing` flips to `ready_for_review`.

## Groundedness Guardrail Contract (directly counters AI-SPEC §1 Failure Mode #5 "rubber-stamp approval")

This is the single most important visual element this phase introduces — AI-SPEC.md Section 6 names it the highest-leverage counter to the phase's #1 named risk. It must not be treated as a decorative nice-to-have.

**Per-item flag chip** (rendered inline, directly under the flagged `advice_point`/`improvement_note` text, never as a separate summary-level banner — the whole point is drawing the eye to the *specific* item):

```
┌─────────────────────────────────────────┐
│ ⚠ Perlu Verifikasi                       │  ← bg-orange-100 text-orange-700
│ Tidak ditemukan tepat di transkrip.       │  ← text-[11px] font-normal, same chip
│ Periksa sebelum menyetujui.               │
└─────────────────────────────────────────┘
```

- Chip: `inline-flex items-center gap-1.5 bg-orange-100 text-orange-700 border border-orange-300 rounded-lg px-2.5 py-1.5 text-[11px] font-bold` with `warning` Material Symbol icon (14px), followed by the same-size `font-normal` explanatory line directly beneath it (not truncated, not a tooltip-only reveal — must be always-visible, since a hidden/hover-only warning defeats the anti-rubber-stamp purpose).
- Applies to **either** signal the backend surfaces per AI-SPEC §6 (the cheap transcript-token cross-check, or the fuller LLM-judge pre-flag, whichever the planner lands on) — the visual treatment is identical regardless of which mechanism produced the flag.
- **Does not block editing or approval** — flags, never blocks (AI-SPEC §6 is explicit: "Flag, do not block").
- Flagged items are visually sorted to the top of their list (advice_points / improvement_notes each sorted independently) so a time-pressured lecturer sees the riskiest items first, not buried at position 4 of 5.

### S-13: Lecturer Review/Edit/Approve Summary (STT-04)

**Route:** `/dosen/logbook/:sessionId` (new)
**Actor:** Approved lecturer (session's own advisee only — ownership check, same pattern as every other lecturer-scoped view)
**Layout:** Mobile-first stacked; desktop (`lg:`) split-pane — reuses the "PDF left / notes right" split-pane pattern already established in Phase 1 S-09 (`D-16`), applied here as **transcript left / editable summary right**.

**Mobile layout (stacked, default):**
1. Header: session date + student name + SessionLogbook status badge
2. **Transcript panel** (collapsed by default): a card headed "Transkrip Sesi" with a "Lihat Transkrip Lengkap" toggle button (`text-primary font-bold text-sm`, `expand_more`/`expand_less` icon). Collapsed state shows only the first ~2 lines with a fade-out gradient (`bg-gradient-to-b from-transparent to-white`) hinting more content exists.
   - Expanded state: `max-h-80 overflow-y-auto` scrollable block, `text-sm font-normal leading-[1.6] text-slate-700 whitespace-pre-wrap`.
   - **Design requirement (anti-rubber-stamp, not optional):** the primary "Setujui & Kunci" button (below) stays visually present but is **disabled** (`opacity-50 cursor-not-allowed`) until the transcript panel has been expanded at least once in this viewing session. Once expanded, it stays enabled even if the lecturer collapses the panel again — this is a one-time nudge, not a persistent gate. Tooltip/`title` on the disabled button: "Buka transkrip lengkap terlebih dahulu."
3. **Advice Points section** — heading "Saran Dosen" (`font-headline font-bold text-lg`). Each item is a card: editable `topic` (bold, `text-sm`) + editable `detail` (textarea, `text-sm`, auto-grow), pencil icon to toggle inline edit mode, trash icon (`text-error`) to remove the item entirely (no confirmation modal — see Copywriting Contract's Undo pattern). Groundedness flag chip (see contract above) renders directly under any flagged item.
4. **Improvement Notes section** — identical card treatment, heading "Catatan Perbaikan untuk Mahasiswa", fields `area` + `action`.
5. **Empty state per section:** if `advice_points` (or `improvement_notes`) is an honestly-empty array (AI-SPEC D5), do not render a placeholder card — render a single muted line: "Tidak ada saran dosen yang tercatat dalam sesi ini." (`text-sm text-neutral-gray italic`) so an empty array reads as a deliberate signal, not a broken UI.
6. Sticky footer CTA: "Setujui & Simpan ke Logbook" — opens S-14 confirmation modal, does **not** commit directly.

**Desktop (`lg:` 1024px+):** two-column grid, transcript panel fixed to the left column (`lg:sticky lg:top-20`, always expanded, no collapse toggle needed since desktop has room) at `lg:max-h-[28rem]`, summary editor in the right column, same sticky-footer CTA full-width under the right column only.

## Accessibility Contract (additions this phase)

- In-progress SessionLogbook badges (`transcribing`/`summarizing`): `role="status" aria-live="polite"` so status transitions are announced without a full page reload.
- Groundedness flag chip: never color-only — icon + "Perlu Verifikasi" text label + explanatory sentence, always visible (not hover/tooltip-gated), satisfying the existing project-wide "color-only status" ban.
- Jitsi video container: `<div role="group" aria-label="Panggilan video sesi bimbingan">` wrapping the embed, so screen-reader users get a labeled region even though the call's own internal controls live inside Jitsi's iframe (outside this app's accessibility tree — a known, accepted limitation of iframe-embedded third-party UI, not something this phase can fix).
- Delete-item toast: `role="status" aria-live="polite"`, "Urungkan" action reachable via keyboard tab order before the toast's own auto-dismiss.
- Approve confirmation modal: `role="dialog" aria-modal="true"` + `aria-label="Konfirmasi Persetujuan Logbook"`, exact same shape as existing `ConsentModal`/`EmergencyCancelModal`.
- All new icon-only buttons (transcript toggle when collapsed to icon-only on narrow viewports, item delete/edit icons): `aria-label` required, matching the existing `SymptomConfig.tsx`/`SessionTable.tsx` precedent (`aria-label="Edit gejala"`, `aria-label="Lihat detail sesi..."`).
