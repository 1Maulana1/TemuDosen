<objective>
Extend the existing Admin Dashboard (not a new page, D-10) with STT/LLM monitoring (S-17, ADMIN-05): job success/failure counts from `SystemLog` filtered by the Phase-6 `event_type` set, and a monthly cost estimate that SUMS the per-session `llm_cost_estimate_idr` values the pipeline already stored (D-11) — the dashboard never re-queries the Anthropic API or recomputes cost O(n) on load. This plan is file-disjoint from the logbook UI plan (06-06), so both run in parallel in Wave 4.

Purpose: Prodi Admin needs pipeline-health visibility and cost tracking. Reusing `SystemLog` (D-10) and pre-stored cost values (D-11) keeps this cheap and scalable, consistent with the phase's async non-blocking discipline.

Output: `AdminStatsView` STT/LLM block + monthly cost sum (+ backend test), `api/stats.ts` new fields, and the "Pemrosesan STT/AI" section in `AdminDashboard.tsx` with the cost card and filtered failure log.
</objective>

<artifacts_produced>
This plan produces:
- **`AdminStatsView` extension:** a `stt_llm` block in the response — `transcription_success`, `summary_success`, `failed_fallback` (aggregate count over event_types `{STT_FAILED, STT_TIMEOUT, STT_EMPTY, STT_DISABLED, LLM_FAILED, LLM_TIMEOUT, LLM_VALIDATION_FAILED, LLM_DISABLED}`), `monthly_cost_idr` (Sum of `SessionLogbook.llm_cost_estimate_idr` for the current month), `avg_cost_per_session_idr` (monthly sum ÷ approved sessions this month)
- **`api/stats.ts`:** the `stt_llm` fields added to the admin-stats response type + a helper to fetch the STT/LLM-filtered failure log via the existing `/api/admin/logs/?type=` endpoint
- **`AdminDashboard.tsx`:** the "Pemrosesan STT/AI" section (4-card grid + cost card + filtered failure log), inserted between "Status Integrasi" and "Emergency Cancel"
</artifacts_produced>
