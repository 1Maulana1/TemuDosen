<objective>
Build the lecturer and student logbook UI: the list entry point (S-12), the review/edit/approve screen with its anti-rubber-stamp gate and groundedness chips (S-13/S-14), the student read-only view (S-15), and the manual-notes fallback (S-18). This plan owns the shared frontend surface (`api/logbook.ts`, `router.tsx` logbook routes, `LogbookStatusBadge`, the `LecturerDashboard` nav item) so the admin plan (06-07) stays file-disjoint and parallel.

Purpose: The pipeline (Waves 2-3) produces summaries; this is where a lecturer actually reviews/edits/approves them (STT-04) — the pipeline's sole safety gate — and where students read approved results (STT-06). The groundedness chip is the single highest-leverage counter to rubber-stamp approval (AI-SPEC FM #5) and must be built as specified, not as decoration.

Output: `api/logbook.ts`, `LogbookStatusBadge.tsx` (+ test), router routes + nav, `LecturerLogbookList`, `LecturerLogbookReview`, `ApproveLogbookModal`, `LecturerManualNotes`, `StudentLogbookView`, and the `SessionTable` extension.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| lecturer eye ↔ AI summary | The human review gate is the pipeline's sole safety mechanism (AI-SPEC FM #5) |
| student browser → logbook API | Students must never see unapproved content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
